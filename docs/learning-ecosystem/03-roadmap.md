# 03. ロードマップ — Phase 全体展望

各 Phase は「回したいループの周期」に対応する。
**Phase 1 = 日次ループ、Phase 2 = 週次ループ、Phase 3 = 月次〜四半期ループ(資産化)**。

前の Phase のループが実際に回り始めてから次に進む。特に Phase 2 の可視化・サマリは、
実ノートが数件〜十数件溜まった状態で実データ検証するのが望ましい。

```
Phase 1 ─ 日次ループを成立させる(捕獲の摩擦をゼロにする)
   │        content/notes/ + /notes ページ + Issue→ノート捕獲Action + scaffolding
   ▼
Phase 2 ─ 週次ループを成立させる(在庫を見せて消化圧をかける)
   │        /learning ダッシュボード + 週次サマリAction(統計+AI傾向コメント)
   ▼
Phase 3 ─ 資産化・定着強化(キャリアへの接続)
            シリーズページ / Aboutスキルマップ連動 / 復習リマインド
```

---

## Phase 1: 日次ループ(捕獲)

**目的**: 破綻点1「記録されないインプット」を潰す。
学んだ瞬間から5分以内に、スマホだけでノートが公開サイトに載る状態を作る。

**このPhaseが解く問題**: オーナーのインプットは移動中・書籍・動画が大半で、
PCの前にいない。捕獲経路がスマホで完結しなければサイクルの入口が成立しない。

| 成果物 | 手段 |
|--------|------|
| ノートのデータ層 | `content/notes/*.md` + `src/lib/notes.ts`(posts.ts のパターンを流用) |
| ノート公開ページ | `/notes`(一覧・タグフィルター・生煮え断り書き)+ `/notes/[slug]`(詳細) |
| スマホ捕獲経路 | GitHub Issue フォーム(テンプレート)→ GitHub Actions が Markdown 化して `main` へ commit → 自動デプロイ |
| PC捕獲経路 | `npm run new:note` / `npm run new:post`(テンプレート入りMD生成スクリプト) |
| ナビ導線 | Header に Notes を追加 |

**完了条件**:
- [ ] スマホの GitHub アプリから Issue を立てると、数分後に https://HyuDev7.github.io/notes に反映される
- [ ] 本人以外が Issue を立ててもノート化されない(セキュリティ要件。04参照)
- [ ] `npm run new:note` で正しい frontmatter のファイルが生成される
- [ ] `npm run build` が通り、既存ページ(blog等)に回帰がない

**詳細仕様**: [04-phase1-spec.md](./04-phase1-spec.md)

---

## Phase 2: 週次ループ(可視化と外圧)

**目的**: 破綻点2「消化されない在庫」と破綻点4「フィードバック不在」を潰す。
ノート在庫をタグ別に見せ、週次で「そろそろこのテーマは記事化できる」と突きつける。

**前提**: Phase 1 が稼働し、実ノートが溜まり始めていること。

| 成果物 | 手段 |
|--------|------|
| 学習ダッシュボード | `/learning` ページ。タグ別パイプライン(ノート件数 vs 記事化済み件数)、直近の活動をビルド時集計で表示 |
| 週次サマリ | GitHub Actions(cron)。①スクリプトによる決定的な統計(今週のノート数・タグ内訳・今月の記事数・昇格候補タグ) ②AI による傾向コメントと記事化の切り口提案(オプション、API キー無しでも①だけで動くこと) |
| サマリの届け先 | リポジトリの Issue として起票(スマホの GitHub アプリに通知が来る = 捕獲経路と同じ場所で受け取れる) |

**完了条件**:
- [ ] /learning でタグごとの「ノート◯件・記事◯件」が見える
- [ ] 毎週決まった時刻に統計 Issue が自動で立つ
- [ ] ANTHROPIC_API_KEY 未設定でも統計部分だけで動作する(AIパートはgraceful degradation)

**詳細仕様**: [05-phase2-spec.md](./05-phase2-spec.md)

## Phase 3: 資産化・定着強化(復習リマインドから着手中)

**目的**: 破綻点3・4の残りを潰しつつ、蓄積をキャリア資産に変換する。
候補A・Bの仕様は Phase 1-2 の運用実感を踏まえて確定する(現時点で作り込まない)。

| 候補 | 概要 | 状態 |
|------|------|------|
| 復習リマインド | 公開からN週間後に「再読・追記」を促す(間隔反復による定着強化) | 実装中。[08-review-reminder-spec.md](./08-review-reminder-spec.md) |
| シリーズページ | 同一テーマの記事群を「学習の軌跡」として束ねる。体系的知識の証明としてポートフォリオ性が高い | 方向性のみ |
| スキルマップ連動 | About のスキルセクションを、実際の記事・ノートのタグ集計と接続する(「Kotlin: 記事5本」のように裏付きにする) | 方向性のみ |
| 英訳パイプライン完成 | 未完成の translate.yml を完成させ、記事を自動で英語資産化する(本エコシステム外の独立タスクだが、L3の価値を大きく上げる) | 独立タスク |

**詳細**: [06-phase3-spec.md](./06-phase3-spec.md)

## Phase をまたぐ不変条件

- どの Phase の変更でも `npm run build` が通ること(= サイトが壊れないこと)
- 既存の blog / projects / about 等の挙動を変えないこと(変える場合はオーナー合意)
- データはすべてこのリポジトリ内の Markdown / JSON で完結させること
- AI関与の境界ルール([01-vision.md](./01-vision.md))に違反する自動化を作らないこと
