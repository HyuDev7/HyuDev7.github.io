# 09. ノートの追記 実装仕様(Phase 1 拡張)

ステータス: **実装済み**(2026-07-08)

Phase 1 稼働後に判明したギャップへの対応: 公開済みノートに後から加筆する手段が
存在しなかった(捕獲は新規作成のみ)。

## 目的

学習は一度で完結しないことが多い(後から気づいた補足、間違いの訂正、関連情報の
追加)。生煮え公開をコンセプトにしている以上、後から育てられる手段が必要。

## 設計方針

- **PC経路**: `content/notes/<slug>.md` を直接編集して push すれば追記できる
  (既存の仕組みで元々可能。新規実装は不要)。
- **スマホ経路**(本命): note-capture.yml が公開時に自動クローズする Issue に、
  **コメントを追加する**とそれが追記としてノートファイルの末尾に反映される。
  クローズ済み Issue でも GitHub はコメント投稿を許可するため、reopen 不要。
- 追記は**本文末尾への追加のみ**(全文置き換えはしない)。frontmatter は一切
  変更しない。

## 反映フォーマット

```markdown
(既存の本文)

## 追記 (YYYY-MM-DD)

(コメント本文)
```

複数回追記すると `## 追記` 見出しが複数並ぶ形になる。

## slug の特定方法

note-capture.yml が公開時に投稿する「公開しました: https://.../notes/<slug>」
コメント(既存の仕組み。新規の隠しマーカー等は追加不要)を Issue のコメント一覧から
探し、URL部分の正規表現 `\/notes\/([a-z0-9-]+)/i` で slug を抽出する。
このコメントは `github-actions[bot]` が1回しか投稿しないため、以降何回追記しても
一意に特定できる。

## 変換 workflow

`.github/workflows/note-append.yml` を新規作成:

- トリガー: `issue_comment: [created]`
- **セキュリティ要件(必須)**: 以下すべてを満たす場合のみ実行する。
  - `github.event.issue.pull_request == null`(PRへのコメントを除外。
    `issue_comment` は PR コメントでも発火するため)
  - Issue に `note` ラベルが付いている
  - `github.event.comment.user.login == 'HyuDev7'`(リポジトリオーナーのみ。
    このガードが無いと第三者のコメントで本番ノートを汚染できてしまう)
  - この条件により `github-actions[bot]` 自身のコメント(公開通知)には
    反応しない(ログイン名が一致しないため)。無限ループの心配はない
- 処理:
  1. Issue のコメント一覧から公開通知コメントを探し slug を特定
  2. `scripts/append-note-comment.mjs` でノートファイル末尾に追記
  3. `github-actions[bot]` 名義で `main` へ commit & push
  4. `gh workflow run deploy.yml --ref main` で deploy を明示的に dispatch
     (04-phase1-spec.md の既知の落とし穴その2と同じ対策が必要)
  5. コメントに 👍 リアクションを付けて完了を知らせる(新規コメント投稿はしない。
     公開通知コメントを一意に保つため)

## 受け入れテスト(実装完了の定義)

1. `scripts/append-note-comment.mjs` をローカル実行し、既存ノートの末尾に
   正しいフォーマットで追記され、gray-matter で問題なくパースできることを確認
2. 公開済みノートの Issue(クローズ済み)にオーナーがコメント → 数分後に
   ノートファイルに追記が反映され、コメントに 👍 が付く
3. オーナー以外のアカウントでコメントしても反応しない
4. `note` ラベルの無い Issue へのコメントや PR へのコメントでは反応しない
5. `npm run build` に回帰がないこと
