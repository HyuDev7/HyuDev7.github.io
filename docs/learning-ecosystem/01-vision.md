# 01. ビジョン — なぜ作るのか

## プロジェクトのそもそもの目的

オーナー(Hugh / HyuDev7)の学習サイクルを、このリポジトリ(公開ブログ基盤)の上で
仕組みとして回すこと。サイクルは次の3段階:

1. **新たな知識のキャッチアップ** — 仕事関連技術の追随と、新分野の開拓
2. **自身の中への消化** — 断片的なインプットを自分の言葉で再構成する
3. **記事執筆による知識の定着** — 書く行為で理解を固定化し、同時に公開資産にする

副次的だが重要な目的として、ここで蓄積される記事・ノート群は
**将来の仕事の獲得機会・キャリアを切り拓くためのポートフォリオ**になる。
「継続的に学び、公開している人」であることの証明を、日々の学習の副産物として得る。

## 設計の出発点: サイクルはどこで壊れるか

この種の個人学習システムが破綻する典型的なポイントを特定し、
それぞれに対策を割り当てるのが本設計の骨格である。

| # | 破綻点 | 症状 | 対策(担当コンポーネント) |
|---|--------|------|--------------------------|
| 1 | 記録されないインプット | 学んだが記録の摩擦が高く揮発する | 軽量TILノート層 + スマホ捕獲経路(Phase 1) |
| 2 | 消化されない在庫 | メモは溜まるが記事にならない。「メモ→記事」のジャンプが大きすぎる | タグ別パイプライン可視化 + 週次サマリによる昇格候補の提示(Phase 2) |
| 3 | 公開ハードルが高すぎる | 完璧主義で寝かせて腐る | 生煮え公開をコンセプト化した /notes(Phase 1)、記事は月1〜2本と割り切る |
| 4 | フィードバック不在 | 公開したら終わりで忘れる | 週次サマリ(Phase 2)、復習リマインド(Phase 3) |

## 4層のパイプラインモデル

```
L0 捕獲        →  L1 消化       →  L2 公開        →  L3 資産化
TILノート         骨子・下書き      ブログ記事         シリーズ / スキルマップ
content/notes/    (作業状態)        content/blog/      (Phase 3)
分単位・毎日      テーマ単位        月1〜2本           四半期単位
```

- **L0**: 5分で書ける断片。frontmatter 最小限。生煮えのまま即公開。
- **L1**: 同一タグのノートが3〜4件溜まったら記事化のシグナル。AIと骨子出しをしてよいが
  本文は書かせない(下記「AI関与の境界」)。
- **L2**: 既存のブログ基盤をそのまま使う。
- **L3**: 記事群をテーマ別シリーズに束ね、About のスキルマップと連動させる。

## 設計原則

1. **摩擦の勾配設計** — 捕獲(L0)は限りなく摩擦ゼロに。フォーマット要求・整形要求を課さない。
   完成度への要求は L2 → L3 と進むにつれて上げる。
2. **重心はノート側** — 記事は月次の収穫であって日々のノルマではない。
   リマインダー類は「急かす」のではなく「在庫を見せる」設計にする。
3. **在庫の可視化が消化圧になる** — /learning ダッシュボードでタグ別の
   「ノート◯件・記事化済み◯件」を常時見せる。
4. **外圧の自動化** — 意思力に頼らず、GitHub Actions が週次で状況を突きつける。
5. **書く行為=定着プロセス** — 執筆を効率化しすぎない。AIに本文を書かせたら
   このシステムの存在意義(定着)が消える。
6. **すべてこのリポジトリで完結** — 外部サービス(Notion等)にデータを分散させない。
   Markdown + Git が唯一のデータストア。可搬性と永続性を最優先。

## AI関与の境界(重要・恒久ルール)

| 工程 | AIの利用 |
|------|---------|
| 捕獲(ノート書き) | 使わない(自分の言葉で書くこと自体に意味がある) |
| 統計・集計 | AI不要。スクリプトで決定的に集計する |
| 週次の傾向コメント・記事化候補の提案 | **使ってよい**(週次サマリActionの定性パート) |
| 記事の骨子・構成の壁打ち | **使ってよい**(ローカルの Claude Code 等で対話的に) |
| 記事本文の執筆 | **使わない。人間が書く**(このシステムの核心ルール) |
| 記事の整形(段落分け・見出し付与・箇条書き化・コードブロック構文・誤字修正) | **使ってよい**。ただし「人間が書いていない文を1文でも追加/言い換えしない」が条件。境界の判定基準は「出力に自分が書いていない文が増えたか」— 増えなければ整形、増えれば執筆(越境)|
| 記事の英訳 | **使ってよい**(公開後の変換であり定着プロセスと無関係) |
