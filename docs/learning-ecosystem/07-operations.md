# 07. 運用ガイド — システム完成後の回し方

このファイルはオーナー(および将来の引き継ぎ者)向けの「使い方」であり、
実装仕様ではない。各 Phase の稼働に合わせて該当セクションが有効になる。

## 日次(Phase 1 稼働後)— 所要 5分/回

**学んだら、その場で捕獲する。** 整形しない。完璧を目指さない。

- 移動中・読書中・動画視聴中(スマホ): GitHub アプリ → このリポジトリ →
  New Issue → 「📝 Note」フォーム → タイトル・タグ・本文を書いて Submit。以上。
  数分後に自動でサイトに公開され、Issue は自動で閉じる。
- PC の前にいるとき: `npm run new:note -- "タイトル"` → 開いて書く → commit & push。

**後から追記したくなったら**:
- スマホ: 公開時に自動クローズされた Issue に**コメントを追加する**だけでよい
  (reopen 不要)。数分後にコメント内容がノート末尾に「## 追記 (日付)」として
  反映され、コメントに 👍 が付く。
- PC: `content/notes/<slug>.md` を直接編集して push する。

**守ること(唯一の規律)**: タグは既存の語彙に合わせる(表記ゆれは /learning の集計を壊す)。
Issue フォームでは既存タグがドロップダウンで出るので、そこにあるものは必ずそこから選ぶ。
「新規タグ」欄は本当に新しいタグにだけ使う(選択肢は自動で同期される)。
PC で直接書くときは /notes の既存タグ一覧を見る。

**やらないこと**: ノートの推敲。ノートへのAI利用(自分の言葉で書くことに意味がある)。

## 週次(Phase 2 稼働後)— 所要 10分/回

日曜夜に「📊 Weekly Learning Summary」Issue が自動で立つ。

1. 統計と AI の傾向コメントを読む
2. 昇格候補タグがあれば「来週書くか、まだ寝かせるか」を決める(決めるだけでよい)
3. Issue を close する(= 振り返り完了のチェックとして機能する)

サマリ Issue が close されずに溜まり始めたら、それは運用が形骸化しているシグナル。
曜日・時刻・内容を見直す。

## 月次 — 記事執筆(消化と定着の本番)— 月1〜2本

1. 昇格候補タグを1つ選ぶ
2. そのタグのノートを /notes で読み返す
3. **骨子出しは AI と壁打ちしてよい**(ローカルの Claude Code 等)。
   ノート群を見せて「この素材で記事の構成案を出して」まではOK
4. `npm run new:post -- "タイトル"` でテンプレート生成
5. **本文は自分で書く**(恒久ルール。ここで手を抜くとこのシステムの存在意義が消える)
6. 書き終えたら**整形は AI に任せてよい**(段落分け・見出し付与・箇条書き化・
   誤字修正・コードブロック構文)。ローカルの Claude Code に下記の
   「整形プロンプト」で依頼する。出力は必ず自分で見直し、
   「自分が書いていない文が増えていないか」を確認してから採用する
7. push すれば自動で公開される

### 整形プロンプト(テンプレート)

Claude Code(または他のAIチャット)に、書き終えた下書きを渡してこう依頼する:

```
このMarkdown下書きを整形してください。やってよいこと:
- 段落分け、見出し(##/###)の追加
- 箇条書き化
- コードブロックの言語指定・構文修正
- 誤字脱字の修正

やってはいけないこと:
- 文章のリライト・言い換え・要約
- 新しい文を追加すること
- 元の文にない情報を足すこと

「自分が書いていない文が1文でも増えたら失敗」という基準で、
機械的な整形だけを行ってください。

---
(ここに下書きを貼る)
```

このテンプレートは特定のツールを前提にしない(スクリプト化していない)。
必要になったらスクリプト化・自動化を検討する(現時点では見送り)。

## AI利用の境界(再掲・恒久ルール)

| してよい | してはいけない |
|----------|----------------|
| 週次サマリの傾向コメント生成 | ノートの代筆 |
| 記事の骨子・構成の壁打ち | 記事本文の生成 |
| 記事の整形(段落分け・見出し・箇条書き化・誤字修正) | 文章のリライト・言い換え・要約(実質的な文章生成) |
| 公開後の英訳 | — |
| 統計集計(そもそもAI不要・スクリプトで) | — |

判定基準: 「AIの出力に、自分が書いていない文が1文でも増えたか」。
増えなければ整形、増えれば執筆(越境)。

理由: 執筆行為そのものが知識定着の手段だから(01-vision.md)。

## トラブルシューティング

| 症状 | 見るところ |
|------|-----------|
| Issue を立てたのにノートが公開されない(job が `skipped`) | ほぼ確実に `note` ラベルが付いていない。GitHub の Issue Forms はラベルがリポジトリに未登録だと黙って付与しない仕様がある(初回のみ起きる問題。一度 `note` ラベルが作成されれば以降は自動で付く)。Issue に手動で `note` ラベルを付けると `labeled` イベントで再処理される |
| Issue を立てたのにノートが公開されない(その他) | Actions タブの note-capture workflow のログ。自分のアカウントで立てたか(他者の Issue は仕様として無視される) |
| note-capture は成功しているのに /notes に反映されない | Actions タブで `Deploy to GitHub Pages` が同時刻に走っているか確認。`GITHUB_TOKEN` での push は他 workflow の `on:push` を発火させない仕様のため、note-capture.yml は commit 後に `gh workflow run deploy.yml` で明示的に deploy を起動している。この dispatch ステップが失敗していないか note-capture のログも確認する |
| サイト全体が更新されない | Actions タブの deploy workflow。`npm run build` がローカルで通るか |
| /learning の集計がおかしい | タグの表記ゆれを疑う(`kotlin` vs `Kotlin` など)。ノートと記事の frontmatter を確認 |
| 週次サマリが来ない | Actions タブの weekly-summary。cron は UTC 表記である点に注意 |
| 週次サマリに AI コメントがない | リポジトリ Secrets の `ANTHROPIC_API_KEY`。未設定でも統計だけは出る(仕様) |
| ノートの Issue にコメントしたのに追記されない | Actions タブの note-append workflow のログ。オーナー本人のコメントか、Issue に `note` ラベルが付いているか確認。公開通知コメント(「公開しました: ...」)が見つからないと slug を特定できず失敗する |
