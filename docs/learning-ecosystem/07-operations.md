# 07. 運用ガイド — システム完成後の回し方

このファイルはオーナー(および将来の引き継ぎ者)向けの「使い方」であり、
実装仕様ではない。各 Phase の稼働に合わせて該当セクションが有効になる。

## 日次(Phase 1 稼働後)— 所要 5分/回

**学んだら、その場で捕獲する。** 整形しない。完璧を目指さない。

- 移動中・読書中・動画視聴中(スマホ): GitHub アプリ → このリポジトリ →
  New Issue → 「📝 Note」フォーム → タイトル・タグ・本文を書いて Submit。以上。
  数分後に自動でサイトに公開され、Issue は自動で閉じる。
- PC の前にいるとき: `npm run new:note -- "タイトル"` → 開いて書く → commit & push。

**守ること(唯一の規律)**: タグは既存の語彙に合わせる(表記ゆれは /learning の集計を壊す)。
迷ったら /notes の既存タグ一覧を見る。

**やらないこと**: ノートの推敲。ノートへのAI利用(自分の言葉で書くことに意味がある)。

## 週次(Phase 2 稼働後)— 所要 10分/回

日曜夜に「📊 Weekly Learning Summary」Issue が自動で立つ。

1. 統計と AI の傾向コメントを読む
2. 昇格候補タグがあれば「来週書くか、まだ寝かせるか」を決める(決めるだけでよい)
3. Issue を close する(= 振り返り完了のチェックとして機能する)

サマリ Issue が close されずに溜まり始めたら、それは運用が形骸化しているシグナル。
曜日・時刻・内容を見直す。

## 月次 — 記事執筆(消化と定着の本番)— 月1〜2本

1. 昇格候補タグを1つ選ぶ
2. そのタグのノートを /notes で読み返す
3. **骨子出しは AI と壁打ちしてよい**(ローカルの Claude Code 等)。
   ノート群を見せて「この素材で記事の構成案を出して」まではOK
4. `npm run new:post -- "タイトル"` でテンプレート生成
5. **本文は自分で書く**(恒久ルール。ここで手を抜くとこのシステムの存在意義が消える)
6. push すれば自動で公開される

## AI利用の境界(再掲・恒久ルール)

| してよい | してはいけない |
|----------|----------------|
| 週次サマリの傾向コメント生成 | ノートの代筆 |
| 記事の骨子・構成の壁打ち | 記事本文の生成 |
| 公開後の英訳 | — |
| 統計集計(そもそもAI不要・スクリプトで) | — |

理由: 執筆行為そのものが知識定着の手段だから(01-vision.md)。

## トラブルシューティング

| 症状 | 見るところ |
|------|-----------|
| Issue を立てたのにノートが公開されない | Actions タブの note-capture workflow のログ。ラベル `note` が付いているか。自分のアカウントで立てたか(他者の Issue は仕様として無視される) |
| サイト全体が更新されない | Actions タブの deploy workflow。`npm run build` がローカルで通るか |
| /learning の集計がおかしい | タグの表記ゆれを疑う(`kotlin` vs `Kotlin` など)。ノートと記事の frontmatter を確認 |
| 週次サマリが来ない | Actions タブの weekly-summary。cron は UTC 表記である点に注意 |
| 週次サマリに AI コメントがない | リポジトリ Secrets の `ANTHROPIC_API_KEY`。未設定でも統計だけは出る(仕様) |

## 引き継ぎ・AIエージェントへの依頼時

- まず `docs/learning-ecosystem/README.md` から読ませる(読む順番が書いてある)
- 実装を依頼する際は「該当 Phase の spec(04/05/06)に従って」と指示する
- 仕様と実装が乖離したら、**同じ変更内でドキュメントも更新させる**
- 設計判断を変えたら 01-vision.md の Decision Log に追記する
